15.交叉编译

# 一、交叉编译的结构基础

# 1. Host 与 Target 定义

  • Host：正在进行编译操作的机器
  • Target：最终运行程序的机器

在当前实操场景中：

  • Ubuntu22.04（x86_64）= Host
  • 香橙派（ARM）= Target

关键问题： 为什么在 Ubuntu 上 gcc test.c 编出来的程序，不能直接在 ARM 上运行？

答案：CPU 指令集不同。

# 2.为什么要进行交叉编译

Target性能太弱，编译巨慢或者根本编译不动，这时候就需要使用更高性能的Host来先把目标文件编译好直接放到Target上面运行

# 3. 指令集架构（ISA）

CPU 不只是“算东西的盒子”，它理解的是特定的指令编码。

  • x86_64 → Intel / AMD 架构
  • ARM → 嵌入式常见架构

同一段 C 代码：

int main()
{    
    return 0; 
}
1
2
3
4

在不同架构上，会被翻译成完全不同的机器码。

机器码不同 → CPU 无法识别 → 无法执行。

# 4. 什么是 ELF？

Linux 下的可执行文件，本质是 ELF 文件（Executable and Linkable Format）。

ELF 头部会明确记录：

  • 架构类型
  • 字节序
  • ABI
  • 入口地址

这就是我们验证 Host / Target 的关键工具。

# 二、交叉编译工具链结构记录

本次不谈概念的 “定义”，只记录结构的确认过程。

环境：Ubuntu 22.04 x86_64

目标：ARM（armhf，hard-float）

# 1. apt 先更新

apt update 的输出本身没有技术含量，但它确认了一件事：

当前系统为 jammy (22.04)，架构为 amd64，源为 USTC 镜像。

这一步的真实作用是：确保 Host 环境干净、可控、可复现。

# 2. 安装交叉工具链：它不是一个 gcc

执行安装命令：

sudo apt install gcc-arm-linux-gnueabihf
1

安装过程中，会出现依赖列表：

  • binutils-arm-linux-gnueabihf
  • cpp-11-arm-linux-gnueabihf
  • gcc-11-arm-linux-gnueabihf
  • libc6-armhf-cross
  • libc6-dev-armhf-cross
  • libstdc++6-armhf-cross
  • linux-libc-dev-armhf-cross
  • libgcc-11-dev-armhf-cross

所谓 “交叉编译器”，本质是：

一整套针对 ARM 架构构建的编译器 + 汇编器 + 链接器 + 标准库 + 头文件 + 启动文件。

gcc 只是入口，真正决定能否链接成功的是那套 ARM 版本的 libc 与开发头文件。

# 3. target triple(目标三元组)

查询交叉编译器路径：

which arm-linux-gnueabihf-gcc
1

输出：

/usr/bin/arm-linux-gnueabihf-gcc
1

名字拆解（目标平台的身份描述）：

  • arm → 架构
  • linux → 操作系统
  • gnueabihf → GNU EABI + hard float
  • gcc → 前端驱动

这不是 “版本名”，而是目标机器的抽象坐标。

# 4. 工具链是家族，而不是单体

查看完整交叉工具集：

ls /usr/bin/arm-linux-gnueabihf-*
1

看到的完整工具集包含：

  • as
  • ld
  • objdump
  • readelf
  • nm
  • strip
  • gcov
  • ar
  • ranlib

这意味着：

交叉编译 ≠ 用一个特殊 gcc

交叉编译 = 整个构建流程都切换到 ARM 版本工具（从预处理到汇编，从链接到分析，全部在 ARM 语境下进行）。

# 5. sysroot 的真实情况

# （1）表面的 sysroot

arm-linux-gnueabihf-gcc -print-sysroot
1

输出：

/
1

表面看没有 sysroot，实际上是使用默认路径。

# （2真正重要的搜索路径

arm-linux-gnueabihf-gcc -print-search-dirs
1

重点路径：

install: /usr/lib/gcc-cross/arm-linux-gnueabihf/11/
libraries: ...
/usr/lib/gcc-cross/arm-linux-gnueabihf/11/
/usr/lib/arm-linux-gnueabihf/
/lib/arm-linux-gnueabihf/
1
2
3
4
5

这才是核心：编译 ARM 程序时，头文件与库文件搜索路径已被完全重定向到 ARM 版本目录。

这就是 “交叉” 的真正含义 —— 不是代码不同，是依赖体系不同。

# 6. 生成 ARM 可执行文件

编译命令：

arm-linux-gnueabihf-gcc test.c -o test_arm
1

验证（file 命令）：

file test_arm
1

输出：

ELF 32-bit LSB pie executable, ARM, EABI5 version 1,
dynamically linked, interpreter /lib/ld-linux-armhf.so.3,
for GNU/Linux 3.2.0, not stripped
1
2
3

关键信息解读：

  • ELF32 → 生成的是 32 位 ARM 程序
  • ARM → 目标架构为 ARM
  • EABI5 → 使用 EABI5 标准
  • hard-float → 使用硬件浮点 ABI
  • 动态链接 → 依赖 ARM 动态链接
