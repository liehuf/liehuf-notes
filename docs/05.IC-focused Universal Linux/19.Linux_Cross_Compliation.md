# 交叉编译工具链结构记录

这一次不谈概念的“定义”，只记录结构的确认过程。

环境：Ubuntu 22.04 x86_64
目标：ARM（armhf，hard-float）

------

## 一、apt 的意义不是更新，而是平台确认

`apt update` 的输出本身没有技术含量，但它确认了一件事：

当前系统为 **jammy (22.04)**，架构为 **amd64**，源为 USTC 镜像。

这一步的真实作用是：确保 Host 环境干净、可控、可复现。

------

## 二、安装交叉工具链：它不是一个 gcc

执行：

```bash
sudo apt install gcc-arm-linux-gnueabihf
```

下载 38.2MB，解压后占用 122MB。

关键观察点不是“安装成功”，而是依赖列表：

- binutils-arm-linux-gnueabihf
- cpp-11-arm-linux-gnueabihf
- gcc-11-arm-linux-gnueabihf
- libc6-armhf-cross
- libc6-dev-armhf-cross
- libstdc++6-armhf-cross
- linux-libc-dev-armhf-cross
- libgcc-11-dev-armhf-cross

结论：

所谓“交叉编译器”，本质是：

> 一整套针对 ARM 架构构建的编译器 + 汇编器 + 链接器 + 标准库 + 头文件 + 启动文件。

gcc 只是入口。

真正决定能否链接成功的是那套 **ARM 版本的 libc 与开发头文件**。

------

## 三、target triple 不是魔法字符串

```bash
which arm-linux-gnueabihf-gcc
```

输出：

```
/usr/bin/arm-linux-gnueabihf-gcc
```

名字拆解：

- arm → 架构
- linux → 操作系统
- gnueabihf → GNU EABI + hard float
- gcc → 前端驱动

这就是目标平台的身份描述。

不是“版本名”，而是目标机器的抽象坐标。

------

## 四、工具链是家族，而不是单体

```
ls /usr/bin/arm-linux-gnueabihf-*
```

看到完整工具集：

- as
- ld
- objdump
- readelf
- nm
- strip
- gcov
- ar
- ranlib

这意味着：

交叉编译 ≠ 用一个特殊 gcc
交叉编译 = 整个构建流程都切换到 ARM 版本工具

从预处理到汇编，从链接到分析，全部在 ARM 语境下进行。

------

## 五、sysroot 的真实情况

```
arm-linux-gnueabihf-gcc -print-sysroot
```

输出：

```
/
```

表面看没有 sysroot，实际上是使用默认路径。

真正重要的是：

```
arm-linux-gnueabihf-gcc -print-search-dirs
```

重点：

```
install: /usr/lib/gcc-cross/arm-linux-gnueabihf/11/
libraries: ...
/usr/lib/gcc-cross/arm-linux-gnueabihf/11/
/usr/lib/arm-linux-gnueabihf/
/lib/arm-linux-gnueabihf/
```

这才是核心。

编译 ARM 程序时：

头文件与库文件搜索路径已被完全重定向到 ARM 版本目录。

这就是“交叉”的真正含义。

不是代码不同，是**依赖体系不同**。

------

## 六、生成 ARM 可执行文件

编译：

```bash
arm-linux-gnueabihf-gcc test.c -o test_arm
```

验证：

```bash
file test_arm
```

输出：

```
ELF 32-bit LSB pie executable, ARM, EABI5 version 1,
dynamically linked, interpreter /lib/ld-linux-armhf.so.3,
for GNU/Linux 3.2.0, not stripped
```

关键信息：

- ELF32
- ARM
- EABI5
- hard-float
- 动态链接
- 解释器为 `/lib/ld-linux-armhf.so.3`

这说明：

1. 生成的是 32 位 ARM 程序
2. 使用 EABI5
3. 使用硬件浮点 ABI
4. 依赖 ARM 动态链接器

------

## 七、readelf 头部确认

```
arm-linux-gnueabihf-readelf -h test_arm
```

关键字段：

- 类别: ELF32
- 系统架构: ARM
- 标志: Version5 EABI, hard-float ABI
- 类型: DYN (PIE)

几点理解：

1. PIE 默认开启，现代 Ubuntu 默认生成位置无关可执行文件
2. hard-float ABI 已在 ELF 标志中写明
3. 入口点存在，但解释器由 ARM 动态加载器提供

ELF 头就是程序的“身份证”。

------

## 八、无法执行是预期结果

执行：

```bash
./test_arm
```

结果：

```
无法执行二进制文件：可执行文件格式错误
```

这不是错误，而是验证。

Host：x86_64
Binary：ARM

CPU 无法解析指令。

到此，闭环完成。

------

## 本阶段真实收获

1. 交叉工具链是完整构建体系，不是单一编译器
2. target triple 是目标平台身份描述
3. ARM 的 libc 与开发环境已被独立安装
4. ELF 头部明确记录架构与 ABI
5. x86 与 ARM 的二进制不可互通
